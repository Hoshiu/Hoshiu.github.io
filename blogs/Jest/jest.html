    关于 Jest 前端自动化测试的感想

    Hoshiu 9/24/2021 Jest

    # 前端自动化测试

    一开始对自动化测试产生兴趣是源于 Leetcode 刷题，经由一位学长推荐，研究了一下自动化测试的东西。

    # 前端自动化测试框架：Jest

    Jest 是一个令人愉快的 JavaScript 测试框架，专注于 简洁明快。

    这些项目都在使用 Jest：Babel (opens new window)、 TypeScript (opens new window)、 Node (opens new window)、 React (opens new window)、 Angular (opens new window)、 Vue (opens new window) 等等！

    Jest 是 Facebook 出品的一个测试框架，相对其他测试框架，其一大特点就是就是内置了常用的测试工具，比如自带断言、测试覆盖率工具，实现了开箱即用。

    而作为一个面向前端的测试框架， Jest 可以利用其特有的快照测试 (opens new window)功能，通过比对 UI 代码生成的快照文件，实现对 React 等常见框架的自动测试。

    此外， Jest 的测试用例是并行执行的，而且只执行发生改变的文件所对应的测试，提升了测试速度。目前在 Github 上其 star 数已经破万；而除了 Facebook 外，业内其他公司也开始从其它测试框架转向 Jest ，比如 Airbnb 的尝试 (opens new window) ，相信未来 Jest 的发展趋势仍会比较迅猛。

    jest 的安装相对简单, yarn add jest即可。

    同时，修改package.json内容如下：

    {
  "scripts": {
    "test": "jest"
  }
}

    这一步是为了引入 Jest，使之能够对本地代码进行测试。

  expect(minus(3,3)).toBe(0)

    # 常用匹配器

    事实上，Jest 的控制台也内置了一些快捷指令以及常用的匹配器

    //	math.test.js
//  测试用例
test('测试 10 与 10 相匹配', () => {
  //  toBe 匹配器 matchers
  //  toBe 适用于简单类型的值进行比较，引用类型由于存放地址不同无法使用其进行比较
  expect(10).toBe(10)
})

test('测试 10 与 10 相匹配', () => {
  //  toEqual 匹配器 matchers
  //  使用 toEqual 对内容进行匹配时，只匹配内容而不匹配引用，因此可以对引用类型进行校验
  const a = { one : 1 };
  expect(a).toEqual({ one: 1 });
})

//  toBeNull()、toBeUndefined、toBeDefined（被定义过，即使是null也可以）
//  toBeTruthy（true）、toBeFalsy（false）

test('not 匹配器', () => {
  const a = 1
  expect(a).not.toBeFalsy();  // not 就是非的意思
  expect(a).toBeTruthy();
})

//  数字相关匹配器
test('toBeGreaterThan 匹配器', () => {
  const count= 10
  expect(count).toBeGreaterThan(9) //  前者的值应大于后者
})

test('toBeLessThan 匹配器', () => {
  const count= 10
  expect(count).toBeLessThan(11) //  前者的值应小于后者
})

test('toBeGreaterThanOrEqual 匹配器', () => {
  const count= 10
  expect(count).toBeGreaterThanOrEqual(10) //  前者的值应大于等于后者
})

//  同理还会有 toBeLessThanOrEqual

test('toBeCloseTo 匹配器', () => {
  const firstNumber = 0.1;
  const secondNumber = 0.2;
  expect(firstNumber + secondNumber).toBeCloseTo(0.3) //  前者的值是否非常接近后者，用于解决浮点数不能精确相等的问题
})

//  String
test('toMatch 匹配器', () => {
  const str = 'http://hoshiu.github.io';
  const secondNumber = 0.2;
  expect(str).toMatch('hoshiu')   //  后面的单词包含在 str 之中
})

//  Array，Set
test('toContain 匹配器', () => {
  const arr = ['hoshiu', 'github'];
  const data = new Set(arr);
  expect(data).toContain('hoshiu');   //  测试数组中是否包含后者
})

//  处理异常
const throwNewErrorFunc = () => {
  throw new Error('this is a new Error');
}
test('toThrow 匹配器', () => {
  expect(throwNewErrorFunc).toThrow('this is a new Error')  //  toThrow 函数中不写入内容则判断报错就行，写入了内容则要判断内容与前者的报错内容是否一致 
})

//  再次 Press t 退出

    # 用例的预处理或后处理

    有时我们想在测试开始之前进行下环境的检查、或者在测试结束之后作一些清理操作，这就需要对用例进行预处理或后处理。对测试文件中所有的用例进行统一的预处理，可以使用 beforeAll() 函数；而如果想在每个用例开始前进行都预处理，则可使用 beforeEach() 函数。至于后处理，也有对应的 afterAll() 和 afterEach() 函数。

    如果只是想对某几个用例进行同样的预处理或后处理，可以将先将这几个用例归为一组。使用 describe() 函数即可表示一组用例，再将上面提到的四个处理函数置于 describe() 的处理回调内，就实现了对一组用例的预处理或后处理：

    describe('test testObject', () => {
    beforeAll(() => {
// 预处理操作
    })

    test('is foo', () => {
       expect(testObject.foo).toBeTruthy()
    })

    test('is not bar', () => {
        expect(testObject.bar).toBeFalsy()
    })

    afterAll(() => {
// 后处理操作
    })
})

    # 异步代码

    异步代码的测试，关键点在于告知测试框架测试何时完成，让其在恰当的时机进行断言。针对几种常见的异步代码形式， Jest 也提供了相应的异步测试语法。首先对于异步回调，向其传入并执行 done 函数， Jest 会等 done 回调执行结束后，结束测试：

    //	asyncHello.js
export const asyncHello= (name, fn) => setTimeout(() => fn(`Hello ${name}`), 1000)

// asyncHello.test.js
import { asyncHello } from './asyncHello.js'

test('should get "Hello world"', (done) => {
    asyncHello('world', (result) => {
        expect(result).toBe('Hello world')
        done()
    })
})

    此外，对于 Promise 控制的异步代码，可以直接在 then 回调中进行断言，只要保证在用例中返回该 Promise 对象即可：

    // promiseHello.js
export const promiseHello= (name) => {
	return new Promise((resolve) => {
        setTimeout(() => resolve(`Hello ${name}`), 1000)
    })
}

// promiseHello.test.js
import { promiseHello } from './promiseHello.js'

it('should get "Hello world"', () => {
    expect.assertions(1); // 确保至少有一个断言被调用，否则测试失败
	return promiseHello('world').then((data) => {
        expect(data).toBe('Hello world')
    })
})

    Jest 也支持 async/await 语法的测试，无需多余的操作，只要在 await 后进行断言即可，和同步测试的写法一致。

    # 测试覆盖率

    当我们在package.json中添加"coverage": "jest --coverage"时，则会运行 Jest内置的测试覆盖率工具istanbul (opens new window)，并且生成一个测试覆盖率的报告，在终端输出一个表格，也可以作为网页打开它。

    # 在前端框架中使用

    Jest 可以搭配Vue、React等多种框架进行自动化测试。

    Jest 的一大特色就是提供了快照测试 (opens new window)功能。首次运行快照测试，会让 UI 框架生产一个可读的快照，再次测试时便会通过比对快照文件和新 UI 框架产生的快照判断测试是否通过。

    # 总结

    总的来说，我是因为 Leetcode刷题与它结缘，它也确实为我在刷题时提供了很多便利。在这里推荐一下 vscode中的Leetcode插件，它可以帮助你在自己的编辑器中连接Leetcode刷题，在这里解决每日一题和一些自己想做的题非常方便，插件自带了测试以及提交功能，非常好用。

    相信前端的自动化测试也有发展的前途，从以下几点非常适合引入：

    • 需要长期维护的项目。它们需要测试来保障代码可维护性、功能的稳定性
    • 较为稳定的项目、或项目中较为稳定的部分。给它们写测试用例，维护成本低
    • 被多次复用的部分，比如一些通用组件和库函数。因为多处复用，更要保障质量

    虽然我目前只用于刷题，但相信它对于前端行业来说会拥有自己的一席之地。
